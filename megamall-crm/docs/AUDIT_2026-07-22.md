# MegaMall CRM — Finance / Delivery / Commission Audit
**Date:** 2026-07-22
**Scope:** Client delivery fee, courier payout, and the seller/manager/team-lead/company commission engine (`internal/orders/financial.go`, `internal/delivery_settings`, `internal/logistics_settings`, `internal/courier_tariffs`, `internal/compensation`, `internal/courier`).
**Trigger:** Follow-up to PR [#85](https://github.com/TagMaga/GiyohDarmon/pull/85) ("Include delivery-fee revenue in commission base"), merged to `main` at `622b6ae`.
**Status:** Audit only for sections C/D below — no further code changed in this doc.

## A. Current money model (as coded today)

Three independent flows, all keyed off `orders.delivery_method` (`normal` | `fast`, legacy alias `express`→`fast`):

| Flow | Who pays | Who receives | Source of truth |
|---|---|---|---|
| **Client delivery fee** | Client | Company (via courier collection) | `delivery_settings` singleton — default `normal_fee = 0`, `fast_fee = 50` TJS (`internal/delivery_settings/model.go`) |
| **Courier payout** | Company (from margin) | Courier | `courier_tariffs` (range/bracket rules) → `courier_profiles.payout_normal/payout_fast` flat fallback (`internal/logistics_settings/resolve.go: ResolveCourierPayout`) — **no global default; must be configured per courier**, enforced at assignment time (`ResolveAssignmentPayout` rejects couriers with no profile/rules) |
| **Commissions** (seller/manager/team-lead/company) | — | Seller/manager/team-lead/company | `internal/compensation` — percentage rates (`seller_rate` 10%, `manager_team_rate` 3%, `manager_personal_rate` 20%, `team_lead_pool_rate` 40%), resolved employee→team→global and frozen per-order in `OrderFinancialSnapshot` |

**Order lifecycle:** `orders.Service.Create` freezes `delivery_fee` (per-product override → `delivery_settings.GetFee` fallback) and `total_amount` (product value, excludes delivery fee — confirmed by `courier_collect_amount = total_amount + delivery_fee - prepayment_amount` in `internal/courier/repository.go`). On delivery, `orders.emitFinancialEvents` (`financial.go`) freezes `courier_payout`, computes `commission_base`, and writes the commission ledger via `compensation.ApplyCommissionRules`. Cash handover (`courier.Service.SubmitHandover`) is the only place multiple delivered orders for one courier are summed — into a single cash total, never into a single priced/commissioned entity. There is no order-bundling/trip/route concept anywhere in the schema or code.

## B. What changed in PR #85

**Before:** `commission_base = total_amount − courier_payout` — delivery-fee revenue was excluded from the commission base entirely. The company/seller earned nothing from the net delivery margin (client delivery fee minus courier payout), even on paid ("fast") deliveries.

**After:** `commission_base = total_amount + delivery_fee − courier_payout` (`internal/orders/financial.go:100`). Delivery-fee revenue net of the courier's payout now flows into the same commission split as product revenue.

**Verification:** worked example — urgent order (delivery_fee 50, courier_payout 30) + simple order (delivery_fee 0, courier_payout 20), order totals summing to 460 → `(460+50)-(30+20) = 460`, 10% seller commission = 46 TJS. Formula is linear/additive, so it is correct per-order without needing to model "orders delivered in one trip" as a single unit. CI (`build-and-test`) passed; merged to `main` at `622b6ae`.

**Scope check — nothing else needed to change:**
- `compensation.ApplyCommissionRules` and `rules_test.go` take `commission_base` as a plain input parameter; they don't compute it, so they're unaffected.
- The doc comment in `financial.go` was updated; the equivalent comment in `compensation/service.go:348` was intentionally left alone (see C.1 — different code path, different semantics).

## C. Findings

### C.1 🟡 `compensation.Preview` (`GET /hr/compensation/preview`) still documents/uses the pre-fix formula
`internal/compensation/service.go:417`: `commissionBase := req.OrderTotal - courierPayout`, with doc comment at `service.go:348` still saying `commission_base = order_total - courier_payout`. This is a **separate code path** from the order-delivery flow fixed in PR #85 — it's a live backend endpoint for previewing commission math, not something `emitFinancialEvents` calls.
- Whether this is actually inconsistent with the new order-time formula depends on `req.OrderTotal`'s caller-supplied semantics: `netRevenue := req.OrderTotal - deliveryFee` (line 414) implies callers are expected to pass a grand total that **already includes** delivery fee, in which case `OrderTotal - courierPayout` is arithmetically equivalent to the new `total_amount + delivery_fee - courier_payout` formula and nothing is actually broken — just confusingly worded relative to the real `Order` model's fields (where `total_amount` excludes delivery fee).
- **Current blast radius is low**: the only frontend caller, `PreviewCalculator.jsx`, is only rendered from `features/hr/pages/HrDashboard.jsx`, which is not reachable from the router (`/hr` redirects to `/owner/teams?tab=compensation` — see C.3). No live UI currently exercises this endpoint.
- **Recommendation:** rename `PreviewRequest.OrderTotal` (or add a doc comment) to make explicit that it must include delivery fee, so a future caller doesn't pass product-only value and silently under-count the commission base the way the real order flow did before PR #85.

### C.2 🟡 `courier_fee_confirmed` ledger event is defined but never emitted
`compensation.EventCourierFeeConfirmed` (`compensation/model.go:142`) exists in the `FinancialEventType` enum but no code path ever writes it — only `EventCourierFeeEarned` is emitted, once, at delivery (`financial.go: emitCourierFeeEvent`). If the product intent is "earned at delivery, confirmed at cash handover" (mirroring how order cash itself is confirmed), that second half of the courier-payout ledger was never built. Not a regression from PR #85, but adjacent to it — anyone auditing "does the ledger fully reflect courier payout state" should know this gap exists.
- **Recommendation:** either implement the `_confirmed` emission at handover-confirmation time, or remove the unused enum value — an unused ledger event type sitting next to money-relevant code invites a future bug where someone assumes it fires.

### C.3 🟢 Orphaned legacy HR delivery-tariff endpoints (pre-existing, confirmed still present)
`web-admin/src/features/hr/api.js` (`fetchActiveTariff`, `fetchTariffs`, `createTariff`, `deactivateTariff`) still calls `GET/POST /hr/tariffs*`, and `TariffCard.jsx` still renders against it — but the backing `delivery_tariffs`/`delivery_tariff_ranges` tables and routes were dropped in migration `00071_remove_legacy_delivery_tariffs.sql`. These calls will 404 if ever hit. Only reachable through the router-orphaned `HrDashboard.jsx` (same reachability gap as C.1), so currently dead rather than broken-in-production. No change made; flagging for cleanup since it's easy to trip over while working in this area.

### C.4 🟢 No global default for courier payout — confirmed intentional, not a gap
`ResolveAssignmentPayout` (`logistics_settings/resolve.go:76`) explicitly **rejects** assigning an order to a courier with no configured `courier_profiles` row or `courier_tariff_rules` ("курьер не настроен тариф выплат"). There is deliberately no fallback like "30 TJS fast / 20 TJS normal" baked into the code — every courier's payout is owner/dispatcher-configured through the existing tariff UI. This matches how the worked example in section B was verified (30/20 TJS were treated as that courier's configured rates, not new code defaults) — no change was needed or made here.

### C.5 🟢 Commission engine, snapshot freezing, and courier/client fee separation are architecturally sound
Rates are frozen per-order (`OrderFinancialSnapshot`) for audit; `company_revenue` is a residual so `company + seller + manager + team_lead_pool == commission_base` exactly by construction; `emitCourierFeeEvent` is idempotency-guarded against double-emission; house orders correctly skip seller/manager/team-lead attribution. No issues found in this part of the engine.
