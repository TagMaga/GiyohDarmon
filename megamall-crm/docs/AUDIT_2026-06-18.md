## B. Current business flow (as coded today)

1. **Order creation** (`orders.Service.Create`): seller/manager/team-lead picks customer, products (price snapshot per line), warehouse, delivery method (`normal`|`express`), optional prepayment + proof upload.
   - `subtotal = Σ(qty × unit_price)`; `total_amount = subtotal` (delivery **not** added to total_amount).
   - `delivery_fee` ← `delivery_settings.GetFee(method)` (singleton normal/express).
   - A **financial snapshot** is built via `compensation.BuildSnapshot`, which **independently resolves a delivery tariff** from `delivery_tariffs` and freezes commission rates.
   - `net_revenue = total_amount − delivery_fee` (commission base).
   - `total_order_amount = total_amount + delivery_fee` (what client pays); `amount_to_collect = total_order_amount − prepayment`.
   - No prepayment → status `confirmed`; prepayment required → `new` awaiting dispatcher verification.
   - Inventory reserved (`reserved_quantity += qty`).
2. **Dispatcher** confirms prepayment (verify/reject), then `confirmed → assigned` (assign courier via `order_assignments`, sets `orders.courier_id` cache). Can unassign (`POST /dispatch/orders/:id/unassign` → back to `confirmed`, releases assignment).
3. **Courier**: `assigned → in_delivery → delivered` (or `returned`/`issue`). On delivery, commission ledger events are emitted (`orders.emitFinancialEvents`) from the frozen snapshot.
4. **Cash handover** (`courier.Service` create handover): per delivered order
   - `collected = total_amount + delivery_fee − prepayment` (cash from client)
   - `returns = collected − delivery_fee` (courier hands product money back, **keeps delivery_fee**)
   - Dispatcher confirms handover on Cash page.
5. **Owner** sees Finance (ledger-based revenue/commissions) and Logistics (courier performance) dashboards.

## C. Problems found

### C.1 🔴 CRITICAL — Courier payout is conflated with the client delivery fee
`internal/courier/service.go:277` → `returns = collected − DeliveryFee`. The courier's earning **is** whatever the client was charged for delivery.
- **Free delivery breaks payout**: if `delivery_fee = 0` (normal delivery free for client), courier keeps `0` → courier is unpaid. Violates the stated rule: *"Normal delivery can be free for the client, but the courier still gets paid from the company margin."*
- Courier payout **cannot** vary by courier, city, delivery type, or manual override — it is structurally tied to the client charge.
- **Fix:** introduce a separate `courier_payout` concept (configurable tariff + per-courier/per-city/per-type overrides), decoupled from `delivery_fee`. See section D.

### C.2 🔴 CRITICAL — Two competing delivery-fee systems (inconsistent)
- `delivery_settings` (singleton, migration `00047`) — `normal_fee`, `express_fee`. **Used for the real `order.delivery_fee`.** `express_fee` default is **hardcoded to 50** (`delivery_settings/model.go:13`).
- `delivery_tariffs` + `delivery_tariff_ranges` (migrations `00008`/`00012`, fixed/tiered, time-bounded) — **used only by `compensation.BuildSnapshot`** to freeze a `TariffFee` in the snapshot.
- **Consequence:** `order.net_revenue` is computed from the *settings* fee, while the snapshot freezes a *different* tariff fee. The two can disagree. Worse, `BuildSnapshot` returns `ErrNoActiveTariff` if no active tariff row exists → **order creation fails** even though the actual fee comes from settings. Two sources of truth for one number.

### C.3 🔴 Dead ledger path — courier earnings never recorded
`EventCourierFeeEarned` / `EventCourierFeeConfirmed` are **defined** (`compensation/model.go:135-136`) and **read** by the logistics dashboard (`logistics/repository.go:453,573`) but are **never written** anywhere (`emitFinancialEvents` never emits them). Owner Logistics "courier earnings from ledger" is therefore always empty/zero. Courier payout exists only implicitly as kept cash, never in the financial ledger.

### C.4 🟠 Duplicate / thin wrapper pages
- `pages/DispatcherDashboard.jsx`, `pages/OwnerDashboard.jsx`, `pages/SellerDashboard.jsx`, `pages/WarehouseDashboard.jsx` duplicate/wrap their `features/*` equivalents (e.g. DispatcherDashboard just renders `features/dispatcher/pages/DispatcherBoard`). Two parallel locations for "pages."
- `features/hr/pages/HrDashboard.jsx` is effectively orphaned from routing (`/hr` redirects to `/owner/teams?tab=compensation`); only referenced from a People component. HR logic now lives under People/Teams — the HR feature folder is legacy.
- Heavy structural duplication across **seller / manager / team-lead** (each has near-identical Income, Orders, Sellers, Dashboard pages). Strong candidate for shared components parameterized by role.

### C.5 🟠 Owner panel gaps vs. business scope
Owner has no: Warehouse view, Reports, global Settings hub (only delivery fee), Couriers section (redirects to employees filter), HR/commission settings surface. Catch-all renders `ComingSoon`. The Owner CRM is the stated modernization target but is currently the thinnest panel.

### C.6 🟡 Status / UX issues
- `total_amount` excludes delivery while `total_order_amount` includes it — two "totals" shown in different panels (dispatcher shows order total, courier shows amount-to-collect). Labeling is correct in code comments but easy to misread in UI.
- `express` delivery is the only "fast" tier; no per-city / scheduled / zone concept.
- No tariff-by-city or zone anywhere despite multi-city logistics.

### C.7 🟢 What is healthy (preserve)
- Status state-machine is explicit and tested (`model.go allowedTransitions`, transition/RBAC/hierarchy tests).
- Commission engine is principled: net_revenue base, residual team-lead pool, frozen snapshots, immutable event ledger.
- Inventory reservation + warehouse module are solid (recent 10/10 pass).
- Zombie-order/unassign bugs already fixed (see project memory).

## D. Delivery tariff redesign proposal

**Principle: separate two independent money flows.**

| Concept | Who pays | Who receives | Source |
|---|---|---|---|
| **Client delivery fee** | Client (on top of product) | Company | Delivery tariff (by type/city) |
| **Courier payout** | Company (from margin) | Courier | Courier-payout tariff + overrides |

These are **never** equal by definition. Example (product 100 TJS):
- Free normal delivery: client pays **100**, courier still earns **20** (from margin) → company keeps `company_margin − 20`.
- Paid delivery (fee 20): client pays **120**, courier earns **20** (or its own tariff value) → fee covers the payout, margin untouched.

### D.1 Data model (proposed)
1. **`delivery_tariffs`** (client-facing) — unify the two existing systems into one:
   - `id, name, delivery_type (normal|express|scheduled), city (nullable=all), client_fee, is_free_for_client (bool), active_from/to`.
   - Tiered ranges remain optional. Retire the `delivery_settings` singleton (migrate normal/express → rows).
2. **`courier_payout_tariffs`** (NEW) — company→courier:
   - `id, name, delivery_type, city (nullable), payout_amount, calc_method (flat|per_km|tiered), active_from/to`.
3. **`courier_payout_overrides`** (NEW) — `courier_id, city?, delivery_type?, amount, active_from/to` for per-courier manual overrides (most specific wins).
4. **Order fields**: keep `delivery_fee` (client) and **add `courier_payout`** frozen on the order at assignment/delivery (resolved from payout tariff + override). Freeze in snapshot like commissions.

### D.2 Resolution order for courier payout
manual override (courier+city+type) → courier override (courier) → payout tariff (city+type) → payout tariff (type) → default. Resolved value frozen at delivery.

### D.3 Ledger & handover changes
- Emit `courier_fee_earned` (and `_confirmed` on handover confirm) using **`courier_payout`**, not `delivery_fee`. This finally populates the Logistics dashboard.
- Cash handover `returns` must NOT subtract courier payout from client cash. Courier returns **all client cash** (`total_amount + delivery_fee − prepayment`); courier payout is settled separately by the company (payroll/ledger), so free delivery still pays the courier. Settlement of payout becomes its own confirmed flow.
- Company margin per order = `net_revenue × company_rate + delivery_fee − courier_payout` (delivery fee income minus payout, plus commission share). Surface this in Finance.

## E. Owner CRM redesign plan (Apple-style)

Single cohesive Owner shell, soft modern, content-first, generous spacing, one accent color, real data only.

| Section | Contents |
|---|---|
| **Dashboard** | Today/period KPIs (revenue, orders, AOV, margin, deliveries, courier payout), trend sparklines, alerts (low stock, stuck orders, unconfirmed handovers). |
| **Orders** | Global order table + drawer (existing OwnerOrdersPage), filters, analytics, status funnel. |
| **Logistics** | Couriers list + profiles, delivery performance, **payout vs fee** breakdown, cash handovers. |
| **Couriers** | First-class section (not a redirect): roster, status, payout config, performance. |
| **Sellers / Teams** | TeamsHub, team & employee profiles, commission configs, performance. |
| **Finance** | Ledger-based revenue, commissions, **company margin = fee − payout**, cash flow, exportable. |
| **Warehouse** | Owner-level inventory/stock value/movements (read of warehouse module). |
| **HR** | Users, roles, compensation configs (absorb legacy HR feature here). |
| **Settings** | Hub: **Delivery tariffs (client)**, **Courier payout tariffs + overrides**, commission rates, cities/zones, general. |
| **Reports** | Cross-section exports (Excel/PDF): finance, logistics, sales, inventory. |

Design system: shared `<Card>`, `<KpiTile>`, `<DataTable>`, `<Drawer>`, consistent typography scale, dark-mode ready, mobile-responsive (already partly present).
